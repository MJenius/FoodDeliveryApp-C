# Review Module Implementation

## Overview
The Review module enables customers to submit ratings and reviews for restaurants after completing orders. This document describes the complete implementation of the Ratings and Reviews module for the Food Delivery Application.

## Architecture

### Design Pattern
- **Dependency Injection**: All components receive dependencies through constructors
- **Repository Pattern**: Data access is abstracted through interfaces
- **Service Layer**: Business logic is isolated in service implementations
- **Record-based Entities**: Immutable review data structure

## Implemented Components

### 1. Review (Entity)
**File**: `src/main/java/edu/classproject/review/Review.java`

```java
public record Review(String reviewId, String orderId, String customerId, int rating, String comment)
```

**Description**: Immutable record representing a customer's review of a restaurant.

**Fields**:
- `reviewId`: Unique identifier for the review (generated by `IdGenerator`)
- `orderId`: Reference to the order being reviewed
- `customerId`: Reference to the customer who submitted the review
- `rating`: Integer rating (typically 1-5 stars)
- `comment`: Text feedback from the customer

### 2. ReviewService (Interface)
**File**: `src/main/java/edu/classproject/review/ReviewService.java`

**Methods**:
- `Review submit(Review review)` - Submit a new review for an order
- `List<Review> listByRestaurant(String restaurantId)` - Retrieve all reviews for a specific restaurant

**Purpose**: Defines the contract for review operations

### 3. ReviewRepository (Interface)
**File**: `src/main/java/edu/classproject/review/ReviewRepository.java` *(newly created)*

**Methods**:
- `Review save(Review review)` - Persist a review
- `Optional<Review> findByOrderId(String orderId)` - Retrieve review for a specific order
- `List<Review> findByRestaurantId(String restaurantId)` - Retrieve all reviews for a restaurant

**Purpose**: Abstracts data access logic, allowing multiple implementations (in-memory, database, etc.)

### 4. DefaultReviewService (Implementation)
**File**: `src/main/java/edu/classproject/review/DefaultReviewService.java` *(newly created)*

**Constructor Dependencies**:
- `ReviewRepository reviewRepository` - Data access layer
- `OrderService orderService` - For validating that orders exist

**Key Methods**:

#### `submit(Review review): Review`
1. Validates that the order exists using `OrderService.getOrder()`
2. Registers order-to-restaurant mapping in the repository
3. Generates a unique review ID using `IdGenerator.nextId("REV")`
4. Persists the review using the repository

#### `listByRestaurant(String restaurantId): List<Review>`
1. Delegates to `ReviewRepository.findByRestaurantId()`
2. Returns filtered list of reviews for the restaurant

#### Private Method: `validateExistingOrder(String orderId): Order`
- Throws `IllegalArgumentException` if order is not found
- Returns the validated `Order` object for use in review registration

### 5. InMemoryReviewRepository (Implementation)
**File**: `src/main/java/edu/classproject/review/InMemoryReviewRepository.java` *(newly created)*

**Data Structures**:
- `Map<String, Review> reviews` - Stores reviews by reviewId
- `Map<String, String> restaurantByOrderId` - Maps orderId to restaurantId (for filtering by restaurant)

**Key Methods**:

#### `save(Review review): Review`
- Stores review in the `reviews` map
- Returns the saved review

#### `findByOrderId(String orderId): Optional<Review>`
- Uses stream to find first review matching the orderId
- Returns `Optional` for safe null handling

#### `findByRestaurantId(String restaurantId): List<Review>`
- Filters reviews by restaurant using the `restaurantByOrderId` mapping
- Returns list of all matching reviews

#### `registerOrderRestaurant(String orderId, String restaurantId): void`
- Internal method for maintaining the restaurant mapping
- Called by `DefaultReviewService` when a review is submitted

## Dependencies

### Internal Dependencies
- **OrderService**: Validates that orders exist before allowing reviews
- **IdGenerator** (from `edu.classproject.common`): Generates unique review IDs with prefix "REV"

### External Interfaces
- Integrates with the Order module for validation
- Does not introduce circular dependencies

## Design Decisions

### 1. Order Validation
Reviews can only be submitted for existing orders. This ensures data integrity and prevents orphaned reviews.

```java
Order order = validateExistingOrder(review.orderId());
```

### 2. Restaurant Mapping Storage
Since the `Review` record doesn't contain `restaurantId`, the repository maintains a separate mapping from `orderId` to `restaurantId`. This allows:
- `findByRestaurantId()` to work efficiently
- Reviews to remain lightweight entities
- Decoupling of restaurant information from review entity

### 3. ID Generation
Review IDs are generated server-side using `IdGenerator.nextId("REV")` to ensure:
- Uniqueness across the system
- Consistent prefixing for debugging
- Independence from client input

### 4. Immutable Reviews
The `Review` record is immutable, ensuring:
- Reviews cannot be modified after submission (data integrity)
- Thread-safe by design
- Clear representation of "historical" data

## Usage Example

```java
// Setup
ReviewRepository reviewRepository = new InMemoryReviewRepository();
ReviewService reviewService = new DefaultReviewService(reviewRepository, orderService);

// Submit a review
Review newReview = new Review(
    null,  // ID will be generated
    "ORD-123",
    "CUST-456",
    5,
    "Great food and fast delivery!"
);

Review saved = reviewService.submit(newReview);

// List reviews for a restaurant
List<Review> restaurantReviews = reviewService.listByRestaurant("REST-789");
```

## Testing Considerations

### Unit Tests Should Cover
1. **Successful Review Submission**: Valid order, review persisted correctly
2. **Invalid Order Validation**: Attempting to review non-existent order throws exception
3. **Review Retrieval by Order**: Finding specific review for an order
4. **Restaurant Review Listing**: Retrieving all reviews for a restaurant
5. **ID Generation**: Each review gets unique ID with "REV" prefix
6. **Immutability**: Review fields cannot be modified after creation

### Integration Tests Should Cover
1. Review submission through service layer
2. OrderService integration for order validation
3. Multiple restaurants with various review counts
4. Edge cases (empty reviews, special characters in comments, rating boundaries)
